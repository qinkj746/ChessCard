# 房间聊天消息去重设计

## 背景

房间内发送一条聊天消息后，发送者界面会显示两条相同内容。服务端一次发送只保存一条消息并发布一个 `CHAT_MESSAGE` 事件；客户端则会分别处理 WebSocket 事件和 REST 发送响应。这两个异步结果包含相同的 `messageId`，但目前只有 WebSocket 处理分支会查重，REST 响应分支会无条件追加，因此当 WebSocket 先到时会产生重复显示。

## 目标

- 同一个 `messageId` 在房间聊天列表中最多出现一次。
- WebSocket 与 REST 响应无论以何种顺序到达，结果都一致。
- WebSocket 暂时不可用时，发送者仍能通过 REST 响应看到发送成功的消息。
- 不改变服务端保存、广播或聊天 API 协议。

## 方案

在 `RoomPage` 的状态层集中封装“按 `messageId` 追加消息”的操作。WebSocket 的 `CHAT_MESSAGE` 处理分支和 `_sendChatMessage` 的 REST 成功分支都调用该操作，不再各自维护不同的追加规则。

该操作检查当前列表中是否已有相同 `messageId`：不存在时追加，存在时保持列表不变。输入框仍在 REST 请求成功后清空；消息是否已由 WebSocket 添加不影响发送成功状态和输入框行为。

不选择“仅依赖 WebSocket”方案，因为连接尚未建立或暂时断开时会丢失发送者的即时反馈。不选择“发送后重新拉取全部消息”方案，因为它会为每次发送增加一次请求和延迟，而稳定的 `messageId` 已足以解决一致性问题。

## 数据流

1. 用户提交文本，客户端发送 REST 请求。
2. 服务端保存消息，生成稳定的 `messageId`，并发布 WebSocket 事件。
3. WebSocket 事件或 REST 响应先到的一方将消息加入列表。
4. 后到的一方携带相同 `messageId`，被幂等追加逻辑忽略。
5. REST 成功后清空输入框；失败时保留原有错误处理。

## 测试

在 Flutter Widget 测试中增加竞态回归用例：构造一个可延迟完成的发送请求，在请求完成前发出包含相同 `messageId` 的 WebSocket 事件，再完成 REST 响应，断言页面只显示一条消息且输入框已清空。

现有“发送消息并清空输入框”和“追加 WebSocket 消息”测试继续通过。修复后运行聊天相关 Widget 测试，再运行完整 Flutter 测试套件和静态分析。

## 非目标

- 不增加按房间号重新进入或刷新后恢复房间的能力。
- 不改变聊天消息持久化、排序和历史加载行为。
- 不处理用户主动快速提交两次产生的两条不同 `messageId` 消息。
