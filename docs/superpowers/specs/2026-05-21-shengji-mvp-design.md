# 升级棋牌 MVP 设计

## 背景

在 `D:\Project\AI\ChessCard` 目录生成一个棋牌项目，后台采用 Java，App 采用 Flutter。第一版聚焦“升级/拖拉机/80 分”玩法，采用 1 名真人玩家加 3 名简单 AI 的本机 MVP，先跑通核心规则与牌桌交互，再为后续多人联网房间预留接口边界。

## 目标

- 生成双端项目结构：`server/` Java Spring Boot 后台，`app/` Flutter 客户端，`docs/` 文档。
- 后台负责牌局状态、发牌、叫主、扣底、出牌、计分和 AI 推进。
- Flutter 负责创建游戏、展示牌桌、叫主、扣底、选牌出牌、显示分数与阶段。
- 第一版可本机运行，玩家能从首页创建一局升级并完成基础流程。
- 规则核心必须有自动化测试，优先覆盖发牌、叫主限制、主牌排序、计分升级。

## 非目标

- 第一版不做账号、登录、支付、战绩、聊天、好友。
- 第一版不做真实联网多人同步，但接口风格按后续 WebSocket/房间系统可扩展设计。
- 第一版 AI 只做合法动作和简单策略，不追求强度。
- 第一版拖拉机、甩牌、毙牌先建模并覆盖基础判断，复杂竞技边界可以后续增强。

## 后台设计

后台使用 Spring Boot，核心规则放在不依赖 Web 的领域层，Controller 只做请求响应转换。

主要领域对象：

- `Card`：牌，由牌面、花色、副牌序号组成。双副牌通过 `deckIndex` 区分。
- `Suit`：黑桃、红心、梅花、方块。
- `Rank`：3 到 A、2、小王、大王。
- `PlayerSeat`：南、北、东、西，南为真人默认座位。
- `GameState`：当前阶段、庄家、主花色、级牌、底牌、玩家手牌、当前轮、闲家分数。
- `DeclarationOption`：当前玩家可叫主的花色候选。
- `PlayAction`：玩家一次出的牌。

后台 API：

```text
POST /api/games
GET  /api/games/{id}
POST /api/games/{id}/declare
POST /api/games/{id}/kitty
POST /api/games/{id}/play
POST /api/games/{id}/ai/step
POST /api/games/{id}/next
```

第一版使用 MySQL 持久化牌局快照，不再使用 H2 或内存仓库存储正式牌局。`GameService` 负责创建和推进游戏，`GameRepository` 保留仓储边界，当前实现由 `JpaGameRepository`、`GameSessionEntity`、`GameSessionJpaRepository` 将 `GameState` 序列化保存到 MySQL 的 `game_session.snapshot_json`。`GameSchemaInitializer` 负责把 `snapshot_json` 修正为 `LONGTEXT`，避免完整牌局快照被截断。规则判断当前拆在 `DeclarationRules`、`ScoreRules`、`ShengjiSorter` 等规则类中，后续复杂出牌规则可以继续拆出专门的出牌规则服务。`AiPlayer` 根据状态选择动作。

## 规则边界

基础设定：

- 4 人，两两对家组队，默认南北一队、东西一队。
- 2 副牌共 108 张，每人 25 张，剩 8 张底牌。
- 分牌为 5、10、K，总分 200，闲家抓分，庄家守分。
- 第一局从 A 打起，级牌为 A。

叫主限制：

- 真人没有大王或小王时，不能叫主。
- 有大王时，只允许选择红色花色：红心、方块。
- 有小王时，只允许选择黑色花色：黑桃、梅花。
- 候选花色还必须满足手牌里有对应花色的当前级牌。
- 例如当前级牌 A：有大王和红心 A、没有方块 A，则只显示红心。
- 例如有小王和黑桃 A，则只显示黑桃。
- 首局无庄家时，先叫主者为庄家。

主牌排序：

当级牌为 A，主花色为红心时，手牌从左到右按以下顺序显示：

1. 大王、小王。
2. 主级牌：红心 A。
3. 副级牌：其他花色 A。
4. 主 2：红心 2。
5. 副 2：其他花色 2。
6. 主花色普通牌：红心 K 到 3。
7. 其他花色牌：按花色分组，每组 A、K、Q、J、10 到 3。

出牌：

- 庄家首攻，之后上一轮赢家首攻。
- 跟牌优先同花色。
- 第一版实现单张、对子、基础拖拉机识别。
- 同类型同长度可比较大小；主牌可毙副牌。
- 甩牌第一版只允许在规则确认全部最大时成功，否则退化为最小单张，后续可增强全量边界。

计分：

- 闲家抓到 80 分及以上上台，否则庄家升级。
- 闲家 0 分：庄家连升 3 级。
- 闲家小于 40 分：庄家连升 2 级。
- 闲家 80 到 119 分：闲家上台，庄家降 1 级。
- 闲家 120 分及以上：每多 40 分多升 1 级，最高 3 级。
- 最后一轮闲家赢时抠底：底牌分乘以单张 2 倍、对子 4 倍、拖拉机 `2 * 长度` 倍。
- 打到 K 级并赢下 K 局时通关，本局结束。

## Flutter 设计

页面：

- 首页：创建游戏，进入牌桌。
- 牌桌页：展示四个座位、当前阶段、主花色、级牌、庄家、闲家分数、当前轮出牌。
- 手牌区：底部横向展示真人手牌，按后台返回顺序排列，支持点选。
- 叫主弹层：显示后台计算出的可选花色，不可叫主时显示等待其他玩家。
- 扣底弹层：庄家选择 8 张底牌。
- 操作区：出牌、取消选择、推进 AI。

视觉方向：

- 功能优先，使用清晰牌桌布局。
- 牌面需要能区分花色颜色，红心/方块为红色，黑桃/梅花为深色。
- 主牌可以用轻量标记突出，但不遮挡牌面。
- 第一版不做复杂动画，保留简单发牌和选中状态。

## 数据流

1. Flutter 调用 `POST /api/games` 创建游戏。
2. 后台洗牌、发牌、生成当前真人可叫主选项。
3. Flutter 根据 `GameStateDto` 渲染牌桌和手牌。
4. 玩家叫主后，后台确定庄家和主花色，并进入扣底阶段。
5. 庄家扣底后进入出牌阶段。
6. 玩家或 AI 出牌后，后台校验动作并更新当前轮。
7. 一轮结束时后台结算赢家和分牌。
8. 一局结束时后台结算升级或上台。

## 错误处理

- 后台对非法叫主、扣底数量错误、非法出牌返回 400，并给出中文错误信息。
- Flutter 显示错误提示并保留当前选择。
- 找不到牌局应返回 404，Flutter 提示牌局不存在或已失效并允许重新创建。
- MySQL 或 Redis 不可用时，健康检查应明确暴露异常状态；牌局快照持久化失败时后台返回错误，客户端保留当前界面并提示稍后重试。

- 主牌排序符合大小王、级牌、2、主花色、其他花色顺序。
- 分牌统计正确。
- 升级结算覆盖 0 分、小光、80 到 119、120 及以上、最高 3 级限制。
- 抠底倍率覆盖单张、对子、拖拉机。

Flutter 测试：

- 牌桌页能渲染创建后的状态。
- 手牌选中和取消选择状态正确。
